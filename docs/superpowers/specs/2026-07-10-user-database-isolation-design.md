# 多用户独立数据库隔离设计

## 目标

把当前“统一密码 + 全局会话 + 全局历史表”的单用户系统，改造成真正多用户系统。A 用户和 B 用户登录后，生成历史、对话历史、开品素材、会话、偏好设置、临时文件和保存文件彼此隔离；后续真实充值、扣费、退款、个人 API Key 和调用审计可以在同一架构上继续扩展。

## 推荐架构

采用“主库 + 用户工作库”的 B 方案：

- 主库：使用当前 Spring `spring.datasource`。只存平台级数据，包括账号、钱包、钱包流水、调用审计、用户数据库索引、个人服务配置索引。
- 用户工作库：每个用户一个 SQLite 数据库文件，默认位于 `user-data/users/{userId}/workspace.db`。只存该用户的业务数据，包括生成历史、对话历史、开品素材、会话列表、用户偏好。
- 用户文件空间：每个用户独立目录，默认位于 `user-data/users/{userId}/files/`，下面再分 `temp-output`、`history-refs`、`kaipin-materials`、`gallery`。

这种分层保留了“业务数据天然隔离”的优点，同时避免把资金流水拆散到很多用户库里，保证后续平台对账、风控、退款、账单导出可以直接从主库完成。

## 数据边界

主库表：

- `app_user`：账号、显示名、密码哈希、角色、启用状态。
- `user_wallet`：用户余额、冻结金额、版本号。
- `wallet_ledger`：所有资金变动流水。充值、赠送、扣费、退款、人工调整都必须写这里。
- `generation_usage_log`：每次模型调用审计。记录用户、任务、模式、模型、预估点数、实际点数、状态、错误信息。
- `user_provider_credential`：未来个人 API Key 或供应商账号配置索引。第一阶段只建框架，不启用明文展示。

用户工作库表：

- `workspace_session`：当前用户自己的会话列表和当前会话。
- `conversation_history`：当前用户自己的聊天记录。
- `generation_history`：当前用户自己的生图/视频历史。
- `kaipin_material`：当前用户自己的开品素材库。
- `user_preferences`：当前用户自己的偏好配置，例如自定义保存目录。

## 访问规则

- 后端所有 `/api/**` 业务接口必须从 `HttpSession` 取得当前登录用户，不能信任前端传入的用户 ID。
- `sessionId` 只能作为“用户内部工作会话 ID”，不能作为权限边界。
- 查询、详情、删除、下载、缩略图、历史参考图都必须在当前用户工作库和文件空间内解析。
- `TaskService` 里的异步任务必须记录 `ownerUserId`，查询任务和停止任务时按当前用户过滤。
- 文件接口不能再按任意绝对路径读取。允许读取的文件必须位于当前用户文件空间，或来自当前用户历史记录中登记过的路径。

## 兼容策略

- 首次启动时自动创建默认 `admin` 用户，密码沿用 `app.auth.password`，避免部署后无法登录。
- 登录接口支持 `{username, password}`，也短期兼容旧前端只传 `{password}` 的方式，默认按 `admin` 登录。
- 旧的全局历史表、素材表暂时不删除。第一阶段可以迁移到 admin 用户工作库；迁移完成后新写入只进用户工作库。
- 旧路径仍尽量可读，但必须经过当前用户权限校验。

## 积分与流水框架

第一阶段只做框架，不真实扣费：

- 提交生成任务时写一条 `generation_usage_log`，状态为 `STARTED`。
- 任务成功时标记 `SUCCEEDED`，失败或取消时标记 `FAILED` / `CANCELLED`。
- 预估点数字段先保留为 0 或按本地规则估算，实际扣费字段先不扣减钱包。
- `wallet_ledger` 表和 `user_wallet` 表先建好，支持后续接入充值、扣费、退款、人工调整。

后续真实扣费时必须满足：

- 扣余额和写流水在同一个主库事务中完成。
- 每次扣费必须关联 `generation_usage_log.id`。
- 支持失败退款或释放冻结金额。
- 流水不可物理删除，只能冲正。

## 实施顺序

1. 增加账号、钱包、流水、调用审计主库模型。
2. 增加当前用户上下文和登录服务，替换统一密码登录。
3. 增加用户工作库管理器，按用户创建和初始化 `workspace.db`。
4. 把会话、历史、开品素材、偏好服务迁移到用户工作库。
5. 把生成、视频、开品、保存、下载、历史接口接入当前用户。
6. 给 `TaskService` 增加 owner 校验。
7. 加入调用审计记录，但暂不真实扣余额。
8. 跑测试、打包、部署前检查旧数据兼容。

## 完成标准

- A 用户创建的会话、历史、素材、偏好、任务，B 用户查不到、删不到、下载不到。
- A/B 用户分别有自己的 `workspace.db` 文件。
- 生成任务有调用审计记录。
- 钱包和流水表存在，接口或服务层可以写入框架记录，但当前不会真实扣费。
- 旧默认密码仍能登录 admin，不影响当前部署进入系统。
