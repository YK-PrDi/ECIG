# 运行与发布验收门禁

> 本文档是项目的强制质量门禁，不是建议清单。

## 1. 判定规则

对业务代码、前端、后端、配置或部署产物的修改，只有满足以下条件才可以判定为“正常运行”：

1. 变更范围已检查，没有误改或覆盖其他未提交内容。
2. 缺陷修复有回归测试，并确认测试在修复前失败、修复后通过。
3. 前端静态检查、浏览器关键流程、Maven 全量测试全部通过。
4. 构建产物成功生成，部署文件与本地文件 SHA-256 一致。
5. 生产服务状态、HTTP、认证接口和日志检查全部通过。
6. 部署前存在可用备份，失败时能够立即回滚。
7. 所有未覆盖风险均被明确写出，不用“应该可以”代替验证。

状态定义：

| 状态 | 含义 | 是否可称为正常 |
|---|---|---|
| `PASS` | 命令已实际执行，退出码为 0，结果符合预期 | 是 |
| `FAIL` | 命令失败、断言失败或行为不符合预期 | 否 |
| `BLOCKED` | 因账号、网络、权限、依赖等原因无法执行 | 否 |
| `NOT RUN` | 尚未执行 | 否 |

## 2. 变更前检查

```powershell
cd F:\java\ele-business-java
git status --short
git diff --check
```

必须确认：

- 已识别工作区原有修改，不回退、不覆盖无关内容。
- 本次修改文件和业务目标一致。
- `.env.local`、服务器 `.env`、密码、Key、Token 不进入 Git 和输出记录。

## 3. 缺陷回归门禁

修复缺陷时必须遵循：

1. 新增最小化回归测试。
2. 在修复代码前运行测试，确认它因目标缺陷而失败。
3. 实施最小修复。
4. 再次运行同一测试，确认通过。
5. 最后运行全量测试，避免局部修复造成其他回归。

不能仅通过阅读代码或手动点击一次来代替回归测试。

## 4. 本地强制测试

### 4.1 前端静态与契约检查

以下命令必须全部退出为 `0`：

```powershell
node tools/check-frontend-syntax.js
node tools/check-app-contract.js
node tools/check-billing-ui.js
node tools/check-category-ui.js
node tools/check-custom-style-grid.js
node tools/check-hard-cancel-persistence.js
node tools/check-ui-regressions.js
node tools/test-frontend-session-modules.js
```

如果某项因本次变更新增了关键业务路径，必须增加对应检查脚本，不能只扩大现有正则而不验证实际行为。

### 4.2 Maven 全量测试

```powershell
$env:JAVA_HOME='E:\Program Files\Eclipse Adoptium\jdk-17.0.17.10-hotspot'
$mvn='E:\IDEA\IntelliJ IDEA Community Edition 2025.2.4\plugins\maven\lib\maven3\bin\mvn.cmd'
& $mvn test
```

通过标准：

- 输出包含 `BUILD SUCCESS`。
- `Failures: 0`、`Errors: 0`。
- 不能因为时间紧张改用 `-DskipTests` 代替此步骤。
- 测试日志中的预期异常可以存在，但必须由测试断言覆盖，不能有未解释的新异常。

### 4.3 本地服务检查

本地服务默认地址：`http://127.0.0.1:5021`。

```powershell
Invoke-WebRequest 'http://127.0.0.1:5021/api/auth/check' -UseBasicParsing -TimeoutSec 5
```

通过标准：HTTP `200`，且启动日志中没有导致上下文退出的异常。

### 4.4 浏览器关键流程检查

先设置烟测环境变量，不要把真实值写入脚本或文档：

```powershell
$env:AI_STUDIO_BASE_URL='http://127.0.0.1:5021'
$env:AI_STUDIO_SMOKE_USERNAME='<测试账号>'
$env:AI_STUDIO_SMOKE_PASSWORD='<从安全位置读取>'
```

然后执行：

```powershell
node tools/check-server-smoke.js
node tools/check-authenticated-smoke.js
node tools/check-canvas-chat-drag.js
node tools/check-multi-task-controls.js
node tools/check-skip-analysis-submit.js
```

当前关键行为至少包括：

- 未登录接口权限正确，登录后核心读取接口可用。
- 画布拖入对话区的引用效果不遮挡、不污染整个界面。
- 多个生成任务可同时显示，取消一个任务不影响其他任务。
- “跳过 AI 分析”会直接发送生成请求，不调用分析接口，不残留按钮锁和全局确认状态。
- 页面没有新的 `pageerror` 或控制台错误。

新增或修改支付、下载、视频、画布、生成、取消、并发、登录等关键流程时，必须补充该流程的浏览器自动化检查。

## 5. 构建门禁

全量测试通过后才允许打包：

```powershell
& $mvn clean package -DskipTests
Get-FileHash target\ele-business-java-1.0.0.jar -Algorithm SHA256
```

`-DskipTests` 只表示避免重复执行已经通过的全量测试，不得在未运行第 4.2 节时单独使用。

必须记录：

- JAR 的 SHA-256。
- 本次需要部署的外置前端文件 SHA-256。
- 构建时间和测试时间。

## 6. 部署门禁

生产地址：`http://123.207.0.125:5020/`。

### 6.1 部署前

- 备份当前 `app.jar` 和本次会替换的每一个外置前端文件。
- 备份文件名必须包含时间戳。
- 上传到临时路径，先校验 SHA-256，再原子替换正式文件。
- 只修改前端时不需要无意义重启后端；修改 JAR 时必须重启服务。
- 不修改、不上传、不打印服务器环境变量和密钥。

### 6.2 部署后基础检查

```bash
systemctl is-active ai-studio
curl -fsS --max-time 20 http://127.0.0.1:5020/index.html >/dev/null
sha256sum /www/wwwroot/ai-studio/app.jar
sha256sum /www/wwwroot/ai-studio/frontend/index.html
journalctl -u ai-studio -n 100 --no-pager
```

通过标准：

- 服务为 `active`。
- 首页 HTTP `200`。
- 服务器文件哈希与本地待部署文件完全一致。
- 日志包含本次启动成功记录。
- 没有本次变更引入的 Bean 创建失败、数据库连接失败、端口冲突或持续异常重启。

### 6.3 生产浏览器与接口检查

```powershell
$env:AI_STUDIO_BASE_URL='http://123.207.0.125:5020'
$env:AI_STUDIO_SMOKE_USERNAME='<生产烟测账号>'
$env:AI_STUDIO_SMOKE_PASSWORD='<从安全位置读取>'

node tools/check-server-smoke.js
node tools/check-authenticated-smoke.js
node tools/check-canvas-chat-drag.js
node tools/check-multi-task-controls.js
node tools/check-skip-analysis-submit.js
```

前端有缓存相关修改时，还必须：

- 确认生产 HTML 已引用新版本资源。
- 使用全新浏览器上下文完成自动化检查。
- 通知使用者执行一次 `Ctrl+F5` 强制刷新。

### 6.4 外部供应商验证

修改模型调用、代理、Key 选择、请求参数、轮询、取消或计费逻辑时，必须额外执行一次受控的真实供应商调用，并核对：

- 请求成功或错误能被准确展示。
- 取消后前后端都停止后续处理。
- 失败或取消不会错误扣费、上传、写成功历史。
- 供应商任务 ID、内部任务 ID和用量记录能够对应。

真实调用涉及费用时，应使用最小数量和最低可用规格。未执行真实调用时，必须明确标记为 `BLOCKED`，不能声称供应商链路已验证。

## 7. 回滚门禁

出现以下任一情况必须停止发布并回滚：

- 服务未进入 `active`。
- HTTP、登录或关键浏览器检查失败。
- 哈希不一致。
- 日志出现与本次修改相关的新错误。
- 数据库迁移、扣费、历史记录或任务状态异常。

回滚后必须重新执行：服务状态、HTTP、认证烟测、日志检查。只有回滚版本也验证通过，才能结束故障处理。

## 8. 验收记录模板

每次发布或重要修复结束时，至少记录以下内容：

```markdown
## 验收记录

- 时间：
- 变更目标：
- 修改文件：
- 修复前失败测试：
- 前端静态检查：PASS / FAIL / BLOCKED
- Maven 全量测试：通过数量，Failures，Errors
- 本地 HTTP：
- 本地浏览器检查：
- 构建产物 SHA-256：
- 生产备份路径：
- 生产文件 SHA-256：
- 生产服务状态：
- 生产 HTTP：
- 生产认证烟测：
- 生产浏览器关键流程：
- 生产日志结论：
- 外部供应商真实调用：PASS / BLOCKED / 不适用（说明原因）
- 遗留风险：
- 最终结论：PASS / FAIL / BLOCKED
```

## 9. 最终结论限制

只有最终结论为 `PASS` 时，才允许对外说明“已经修复并正常运行”。

只要存在 `FAIL`、`BLOCKED` 或 `NOT RUN`，最终答复必须准确说明未完成项及风险，不得使用模糊表达掩盖验证缺口。
